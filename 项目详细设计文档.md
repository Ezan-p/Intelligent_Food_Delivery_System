# 智能外卖管理平台项目详细设计文档

## 1. 项目概述

### 1.1 项目名称
智能外卖管理平台

### 1.2 项目定位
本项目是一个基于 `Flask + HTML/CSS + JavaScript + JSON 文件持久化` 实现的多端外卖平台演示系统，面向毕业设计和原型验证场景。系统将平台拆分为三个独立门户：

- 用户端：面向普通消费者，用于浏览店铺、下单、收藏、评价、复购、地址管理与智能服务使用
- 商家端：面向入驻商家，用于店铺管理、商品分类管理、套餐管理、订单处理、经营分析
- 管理端：面向平台管理员，用于统一进行用户管理、商家管理、店铺经营管理、平台概览与智能服务监控

### 1.3 项目目标
项目核心目标包括：

- 构建一个完整的三端分离外卖平台原型
- 实现用户、商家、管理员三类角色的权限隔离
- 支持多商家店铺、多商品分类、多套餐结构
- 支持订单全流程、评价系统、收藏与复购功能
- 引入 AI 客服、智能点餐助手、数据智能分析等智能服务模块
- 通过 JSON 文件实现轻量级数据持久化
- 通过三个端口模拟真实平台中不同门户的访问入口

### 1.4 运行入口与端口设计
项目通过三个端口分别承载不同门户：

- 用户端：`http://localhost:5001`
- 商家端：`http://localhost:5002`
- 管理端：`http://localhost:5003`

未登录时，三端都会自动跳转各自的 `/login` 页面。登录后再进入对应主界面。

## 2. 技术架构设计

### 2.1 技术栈

后端：

- Python 3
- Flask
- requests

前端：

- HTML 模板渲染
- 原生 JavaScript
- CSS 响应式布局

数据层：

- JSON 文件持久化
- 本地文件路径：`data/app_data.json`

智能服务层：

- 远程大模型 HTTP API
- 本地规则回退策略

### 2.2 总体架构
项目整体采用“单后端、多门户、多角色”的轻量架构：

1. `app.py` 提供统一的 Flask Web 服务和 REST API
2. `run_portals.py` 以不同 `PORT` 和 `PORTAL_ROLE` 环境变量启动 3 个服务实例
3. 前端页面按角色分别加载不同模板和脚本
4. 后端依据端口识别当前门户角色
5. 统一数据存储在 JSON 文件中
6. 所有增删改操作执行后立即持久化到本地文件
7. 智能模块优先调用远程 AI，失败时自动回退本地逻辑

### 2.3 三端隔离方式
三端隔离不是通过三个独立后端代码库实现，而是通过同一套后端逻辑做以下区分：

- 基于端口识别当前门户角色
- 基于角色对应不同页面模板
- 基于会话 cookie 名称隔离不同端登录状态
- 基于 API 访问时的用户角色进行权限校验

对应配置定义在后端常量中：

- `PORTAL_PORTS`
- `SESSION_COOKIE_NAMES`
- `ROLE_LABELS`

### 2.4 数据同步策略
由于三个端口对应三个 Flask 进程，项目为了实现商家端修改后用户端自动实时感知，采用了“文件修改时间 + 请求前重载”的方式：

- 所有数据最终落盘到 `data/app_data.json`
- 每个请求进入前执行 `refresh_runtime_data_from_disk()`
- 若磁盘文件的修改时间新于当前进程记录的时间，则重新加载最新数据

这保证了：

- 商家端更新店铺、商品、套餐后，用户端再次请求接口时能拿到最新数据
- 用户端配合前端定时轮询 / 自动刷新逻辑后，可以实现“不手动刷新也能自动更新”

## 4. 系统角色与权限设计

### 4.1 用户角色
系统定义三类角色：

- `customer`：普通用户
- `merchant`：商家用户
- `admin`：管理员

### 4.2 权限设计原则

1. 用户端只能访问自己的地址、订单、收藏、评价、浏览记录等数据
2. 商家端只能访问自身店铺及其商品、分类、套餐、订单
3. 管理端可查看所有用户、店铺、订单，并执行平台级管理动作
4. 未登录用户不能直接查看门户内容
5. 被禁用账户不能继续访问受保护资源

### 4.3 登录与会话机制
会话机制由以下几个部分组成：

- 登录成功后生成 `session_id`
- 保存在内存字典 `user_sessions`
- 同时写入对应门户的 cookie
- 前端也会将 `session_id` 保存到 `localStorage`
- API 请求通过 `X-Session-ID` 请求头或 cookie 识别用户身份

不同门户使用不同 cookie 名称：

- 用户端：`customer_portal_session`
- 商家端：`merchant_portal_session`
- 管理端：`admin_portal_session`

### 4.4 默认账号
系统会自动补齐默认账号：

- 管理员：`admin / admin123456`
- 商家 1：`merchant / merchant123`
- 商家 2：`merchant2 / merchant123`

默认商家账号会自动绑定对应店铺。

说明：

- 普通用户和商家共用一张逻辑用户表

- 一个商家用户 对应 一个店铺
- 一个店铺 包含 多个分类
- 一个分类 包含 多个菜品
- 一个店铺 包含 多个套餐
- 一个套餐 包含 多个菜品
- 一个用户 可以有 多个订单
- 一个订单 属于 一个店铺
- 一个订单 最多对应 一个评价
- 一个用户 可以收藏 多个店铺与多个菜品
- 一个用户 可以记录多条最近浏览记录

## 6. 后端模块设计

后端全部集中在 [app.py](/Users/ezan/Desktop/毕设/Intelligent%20Food%20Delivery%20System/app.py)，可以拆分为以下模块。

### 6.1 基础配置模块
负责：

- Flask 初始化
- 缓存禁用响应头设置
- 数据文件路径定义
- 远程 AI 配置加载
- 角色、端口、cookie、状态集合定义

关键点：

- 所有响应统一加 `no-store`，避免多门户缓存导致页面展示旧数据
- AI 支持通过环境变量配置 API 地址、模型、超时等参数

### 6.2 数据加载与持久化模块
核心函数：

- `load_data()`
- `save_data(data)`
- `persist_data()`
- `update_counters()`
- `refresh_runtime_data_from_disk(force=False)`

设计思路：

1. 启动时加载 `app_data.json`
2. 如果文件不存在则使用默认数据
3. 运行时所有修改直接更新内存列表
4. 修改后调用 `persist_data()` 立即写回磁盘
5. 多进程情况下通过文件修改时间进行同步

### 6.3 数据规范化模块
核心函数：

- `normalize_users()`
- `normalize_stores_and_relations()`
- `ensure_default_accounts()`

功能：

- 为旧数据补齐新字段
- 自动为商家创建店铺
- 自动修正 `store_id` 关系
- 为默认商家补全示例店铺、分类、菜品、套餐数据
- 为记录补上 `status` 字段

该模块的作用是保证项目即使在多次迭代后，旧版本数据也能兼容新版逻辑。

### 6.4 门户识别与导航模块
核心函数：

- `get_request_port()`
- `get_active_portal_role()`
- `build_portal_url()`
- `get_portal_navigation_urls()`
- `render_portal_template()`
- `require_portal_page_access()`

功能：

- 根据当前访问端口推断当前门户角色
- 自动生成跨门户跳转链接
- 校验页面访问权限
- 未登录时跳转到对应端口的登录页

### 6.5 认证与会话模块
核心函数：

- `create_session(user)`
- `get_session_payload(session_id)`
- `get_authenticated_user(required_roles=None)`
- `with_session_cookie(response, role, session_id)`

功能：

- 登录后生成会话 ID
- 维护内存中的会话表
- 同时支持 `X-Session-ID` 与 cookie 获取身份
- 统一做角色校验、状态校验与错误返回

## 7. 前端模块设计

### 7.1 用户端页面设计

#### 7.1.1 首页结构
首页采用“顶部 + 分类快捷入口 + 店铺卡片流”的结构：

顶部区域：

主体内容区：

#### 7.1.2 店铺详情页结构

## 10. 关键业务流程设计

### 10.1 用户下单流程
1. 用户登录用户端
2. 进入首页查看店铺列表
3. 选择某个店铺进入详情页
4. 浏览分类、菜品、套餐
5. 加入购物车
6. 填写姓名并提交订单
7. 后端校验店铺、商品、套餐有效性
8. 生成订单，状态初始化为 `已接单`
9. 持久化到 JSON 文件

### 10.2 商家处理订单流程
1. 商家登录商家端
2. 进入订单列表页
3. 根据状态页签筛选订单
4. 查看右侧订单详情
5. 推进订单状态：已接单 → 制作中 → 配送中 → 已完成
6. 后端校验状态迁移是否合法
7. 更新订单状态并持久化
8. 用户端后续自动获取最新状态

### 10.3 用户评价流程
1. 用户查看自己的订单
2. 对已完成订单点击评价
3. 填写评分、文字、图片
4. 提交后端
5. 后端校验订单归属、状态、是否已评价
6. 写入 `reviews`
7. 用户端店铺评价页可展示该评价

### 10.4 管理员管控用户流程
1. 管理员登录后台管理端
2. 打开用户管理页面
3. 查看用户信息、风险状态、管理员备注
4. 执行禁用/恢复或异常标记
5. 后端写入用户对象
6. 若被禁用，用户后续接口访问会被拦截

### 10.5 智能服务调用流程
1. 前端发起 AI 功能请求
2. 后端组装提示词和上下文
3. 调用远程模型 API
4. 若成功则返回远程 AI 回复
5. 若失败则走本地回退逻辑
6. 同时记录服务监控指标
7. 管理端可查看请求数、失败数和最近调用时间

## 11. 数据持久化设计

### 11.1 持久化策略
所有以下操作都会立即调用 `persist_data()`：

- 注册用户
- 编辑店铺
- 增删改分类
- 增删改菜品
- 增删改套餐
- 创建订单
- 更新订单状态
- 收藏/取消收藏
- 新增浏览记录
- 新增评价
- 编辑地址
- 管理员修改用户或店铺信息

### 11.2 持久化优点
- 实现简单
- 适合毕业设计演示
- 易于查看和调试数据
- 不依赖外部数据库环境

### 11.3 持久化局限
- 不适合高并发
- 图片以 base64 存储时文件会迅速变大
- 不支持复杂事务
- 多进程并发写入存在理论冲突风险

### 11.4 未来可升级方向
如果继续工程化，可迁移到：

- SQLite：适合轻量本地部署
- MySQL / PostgreSQL：适合真实多用户场景
- Redis：可用于会话与缓存加速

## 13. 存在的局限与改进建议

### 13.1 当前局限
- 会话仅保存在内存中，服务重启后登录状态丢失
- 密码为明文存储，不适合生产环境
- 数据层仍是 JSON 文件，不适合高并发与复杂查询
- 管理端前端交互反馈仍有部分原生 `alert/prompt` 风格残留，可继续统一
- 缺少自动化测试体系与接口测试体系
- 缺少日志分级、异常追踪和审计记录

### 13.2 建议的下一步优化
- 引入数据库和 ORM
- 密码改为哈希存储
- 会话改为数据库或 Redis
- 增加分页、排序、筛选能力
- 增加真正的消息中心和用户通知系统
- 增加优惠券、满减、配送员等更完整的平台角色
- 为管理端和用户端继续统一状态反馈组件
- 补充测试用例、部署文档和接口文档
